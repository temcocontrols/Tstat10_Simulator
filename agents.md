# Agent / contributor instructions (Tstat10_Simulator)

Concise rules distilled from recent layout and UI work. Prefer these over improvising conflicting patterns.

1. **Single LCD typeface (hardware parity)**  
   Treat the product as **one** heavy blocky monospace LCD font everywhere: titles, menu rows, IP/values, and footer keys. Keep `styles.fontFamily`, widget-level `font`, and inline HTML `font:` aligned with that stack (e.g. `'Fira Mono', 'Consolas', …, monospace` with strong weight). Avoid mixing `system-ui` or unrelated sans stacks on screens that should match the thermostat reference.

2. **Footer keys in the hierarchy**  
   Standard BACK / ▼ / ▲ / NEXT rows should appear under one tree parent: a logical-only widget `type: "box"`, `id: "footer_buttons_group"`, `treeName: "Footer buttons"`, with each button setting `"parentId": "footer_buttons_group"`. `box` / `panel` / `main_icons_group` are the parents that accept `parentId` children (`widgetAcceptsChildren` in `lcd-editor-core.js`).

3. **Adding a generic tree folder**  
   Use **Widgets → Tree folder** (`data-widget-add="hierarchy_group"`) to insert another invisible `box` for nesting; set children’s `parentId` to that box’s `id`. The **Param row** tile adds a `menu_row`, not a hierarchy-only container.

4. **Layout editor — three surfaces (tree, LCD, inspector)**  
   In **layout edit mode**, the same screen JSON (`window._currentScreenData`) is edited from **three coordinated places**; treat them as one product surface, not three disconnected UIs.  
   - **Hierarchy tree** (`#layout-tree-panel`, `#layout-tree-content`): navigate the widget tree (SquareLine-style expand/collapse, indentation). **Single-click** a row to select (`window._layoutSelectedNodeId`, ids like `w-N`, `w-N-label`, `w-N-value`). **Double-click** or **F2** on a selected widget renames **`treeName`** (in-tree display name) when the widget’s hierarchy row is present; otherwise **F2** falls back to **LCD inline text** rename (`_startInlineEdit`) if that node supports it. Drag-and-drop reorders / reparents where supported.  
   - **LCD** (`#tstat-lcd-container`): **single-click** selects the widget under the cursor (`data-tree-node-id` on elements must stay aligned with tree ids). **Drag / resize handles / context menus / wheel** adjust geometry and some text **on the glass**; **`enableInlineTextEdit`** (headers/labels) and row-specific flows handle inline copy where wired.  
   - **Device shell** (LCD edge strips, hardware keys, reference photo): same selection contract via **`data-tree-node-id`** and **`installTstatShellSelectionSync`** / LCD edge handlers — selecting on the device updates the same id as the tree.  
   - **Inspector** (`#layout-props-panel` → `#layout-props-content`): shows **typed fields for the currently selected node** (`renderLayoutPropertiesPanel` in [`network-settings-renderer.js`](network-settings-renderer.js)). The hierarchy’s flat node list and display titles come from [`layout-tree-model.js`](layout-tree-model.js) (`buildPageNodes`, `findLayoutTreeNodeById`, `getWidgetTreeName`). The **Screen** strip (`#layout-props-screen-section`) holds **page-level** canvas/theme controls (preset, orientation, grid, background, firmware color theme) and stays relevant alongside per-widget rows.  
   **Selection contract:** one canonical id in **`window._layoutSelectedNodeId`**. **Whenever the selection changes** (from tree, LCD, or shell), **`selectLayoutNode`** immediately runs **`renderLayoutTreePanel`** + **`renderLayoutPropertiesPanel`** so the **hierarchy highlight** and **inspector** always match the new object without waiting for a full LCD rebuild. **`applySelectionOutline()`** updates the on-device highlight. **`selectLayoutNode`** also **debounces** a full **`renderScreen`** (~320ms) when the selection **changes** so the LCD DOM is not torn down between the two clicks of a **double-click** (tree rename, LCD inline text). Same-id selection only refreshes **`applySelectionOutline()`**. Starting inline edit cancels the pending timer via **`cancelDebouncedSelectLayoutRender`**. Do not leave stale tree/LCD/inspector state after a selection change.

5. **Inspector refresh (“flash”) & floating panels**  
   Keep `#layout-props-panel` (and related floaters) above overlapping UI (`z-index` / stacking) so fields stay usable in edit mode. **When selection changes**, `renderLayoutPropertiesPanel` **rebuilds** the inspector body (replace children / new DOM). That is intentional and cheap; authors (and you) will often see a **brief flash** as the panel swaps to the newly selected widget. **This is expected behavior**, not a spurious regression, unless you are explicitly implementing incremental/diff updates to the inspector. **Do not leave stale fields** for a previous selection after changing selection — the rebuild is what guarantees correctness. See **item 4** for the three-surface model.

6. **LCD interaction in edit mode**  
   Preserve behaviors such as canvas context-menu widget pick, panning/drag on empty chrome, and documented setup-entry gestures (e.g. L/R hold) without breaking hit-testing under scaled bezel transforms.

7. **Device shell / zoom**  
   Bezel transform, fit, pan, and export paths should stay in sync with `ui-bridge` / zoom so coordinates and overlays remain accurate.

8. **Debug alignment**  
   **Snap grid** is controlled from **Inspector → Screen** (`#toggle-grid-layer`, `window._tstatShowGridLayer`). It overlays the LCD with a row/column grid; colors are tinted from `styles.bg`, not a fixed gray (`lcdGridOverlayColorsFromBg` / `injectLcdSnapGridOverlay` in [`layout-editor-canvas-grid.js`](layout-editor-canvas-grid.js)). Programmatic sync of the checkbox uses `window._tstatSuppressGridToggleRender` so `debug-toggles.js` does not double-fire `renderScreen`. **Coords**, **redbox**, and **auto-tester** stay in the **Debug** panel; changes to row geometry should remain compatible with redbox math.

9. **Persistence**  
   Screen JSON drafts live under **`localStorage`** keys **`tstat_cache_<path>`** (see `saveScreenToCache` / `loadScreenFromCache` in [`visual-edit-local-storage.js`](visual-edit-local-storage.js); project-wide background/theme propagation lives there too). The simulator **loads that cache first** when present so Inspector changes (background, theme, canvas fields, **`canvasProfile.previewOffset`** for LCD workbench nudge) survive reload, navigation, and toggling layout edit off — then **fetches** the repo file only if there is no cache entry. Prefer the save server (**`npm run save-server`**, POST port **5001**) to write JSON to disk; to force a clean pull from shipped JSON, clear the relevant `tstat_cache_…` keys (or site data) in the browser. Workbench LCD translate is mirrored into **`canvasProfile.previewOffsetX/Y`** on each render (**`syncWorkbenchNudgeIntoScreenDataForCache`**) so drafts stay aligned with the **global** nudge store (**item 20**); per-screen cached offsets are **not** reapplied on navigation (that used to jump the LCD between routes).

10. **Authoring direction**  
   Screen definitions are JSON for the web simulator first; the product path is JSON → binary blob on device (not SquareLine-style C export). Keep schemas and `page` ids consistent with `screen_inventory.md`.

11. **LCD row grid vs pixel nudging (recurring alignment issue)**  
   **Problem (comes up often):** Editors assume they can “line up” widgets with small arbitrary vertical moves. On this product the LCD is authored on a **fixed row band grid** (default **32px** per row for a **10×320**-tall logical canvas, matching stock screen JSON and `TSTAT10_MENU_ROW_PX_DEFAULT` in `tstat10-firmware-display.js`). Vertical drag and offsets must snap to that pitch, not to the horizontal quarter-character step.  
   - **Implementation:** `network-settings-renderer.js` uses `lcdRowSnapPx` from `data.layout.menuRowPixelHeight` or **`TSTAT10_MENU_ROW_PX_DEFAULT` (32)**. Floating `label` drags, offset drags, and menu/blank row geometry all use this for **Y**; **X** stays on the character/quarter-cell grid. **Shift** while dragging gives **1px** vertical steps when fine adjustment is needed.  
   - **Footer row:** Buttons that share the same `lcdRow` and `parentId` (e.g. `footer_buttons_group`) move **together**. In layout edit, select a footer button, then drag the **narrow left grip** on any of the four keys, or **Alt+drag** on the button body. **Shift** during the drag disables row snapping (1px steps).  
   - **JSON (optional):** set row pitch explicitly if it ever diverges from the default:  
     `"layout": { "menuRowPixelHeight": 32, ... }`  
   Do not reintroduce “vertical snap = quarter char width” for free-positioned elements; that misaligns the UI relative to the hardware row grid.

12. **Firmware ground truth (ESP32 — `T3-programmable-controller-on-ESP32`)**  
   Treat that repo as authoritative for **panel IC**, **pixel format**, **resolution**, and **stock theme colors**. The web simulator mirrors it via [`tstat10-firmware-display.js`](tstat10-firmware-display.js) and [`lcd-platform-presets.js`](lcd-platform-presets.js) (production preset **240×320 — ILI9341**).  
   - **Driver / bus:** `main/lcd.c` — `ILI9341_Initial()`, GPIO bit-bang SPI (`LCD_IO_Init`). Color depth: register **0x3A = 0x55** → **16-bit RGB565**. **MADCTL** **0x36 = 0xC8** sets scan/mirroring for the mounted panel.  
   - **Framebuffer window:** `ClearScreen` uses `LCD_SetPos(0, 239, 0, 319)` and fills **240×320** pixels — logical **portrait 240 wide × 320 tall** for authoring (matches `TSTAT10_LCD_WIDTH` / `TSTAT10_LCD_HEIGHT`).  
   - **Palette / theme:** `main/LcdTheme.h` — **RGB565** application colors (background **0x7E19**, highlight **0x3cef**, primary text white). CSS mirrors: `TSTAT10_FW_*` in `tstat10-firmware-display.js`. Inspector **Color theme** (item **19**) applies the full **`ThemeList[]`** from `lcdTheme.c` as canned bg + highlight pairs.  
   - **Home row geometry:** `main/LCD_TSTAT.h` / `main/lcd.h` — e.g. `CH_HEIGHT` **36** for large home rows; menu/list layouts may use smaller fonts (`FORM15X30`, etc.). Do not assume **48px** row pitch unless JSON explicitly uses it.

13. **Agent dev environment & console log (automate as much as possible)**  
   - **New humans:** read **[`GETTING_STARTED.md`](GETTING_STARTED.md)** — **`npm install`** → **`npm run setup`** (once) → **`npm start`**. Hooks and CI handle most lint/smoke checks automatically.  
   - **Start substantive simulator work** from the project root with **`npm start`** (`tools/dev-server.mjs`, default **http://127.0.0.1:8787** on the PC). The server binds **`HOST=0.0.0.0`** by default so other devices use **`http://<PC-LAN-IP>:8787/Tstat10.html`**; **`127.0.0.1` on a phone is wrong** (means the phone itself → connection error **~-102**). Set **`HOST=127.0.0.1`** only for local-only. Supported environment: ES modules, **POST /log**, **GET /__tstat_dev_probe** (auto console pipe in `console-pipe-client.js`). Avoid `python -m http.server` for agent-driven debugging unless the user insists.  
   - **After each self-test or automated check** (Playwright/e2e, `npm test` if added, in-app **Auto test**, or any browser exercise while the dev server runs), **read `logs/browser-console.log`** (tail recent lines or full file) and use it in triage. If missing/empty, note that the dev server likely was not used or no logs were emitted.  
   - Cursor picks this up from **`.cursor/rules/tstat10-dev-and-logs.mdc`** (`alwaysApply: true`) for this project.

14. **LCD driver preset vs Apply layout (immediate preview + rescale)**  
   - **Simulator navigation (see item 20):** Every **`renderScreen`** run calls **`enforceSimulatorFixedLcdCanvas`**, which resets the in-memory screen to **production 240×320 portrait** and **`lcdPresetId: 'tstat10_fw_240'`**. The **LCD & driver** / **orientation** controls still update JSON during a session, but the **live** simulator framebuffer slot stays fixed so all routes match hardware — do not expect a larger dev-kit preview in the hosted sim until that rule is relaxed.  
   - **Preset / canvas change:** Changing the **LCD & driver** dropdown or **orientation** (portrait vs landscape) updates `canvasProfile` width/height: catalog presets use `canvasPixelSizeForPresetOrientation` in [`lcd-platform-presets.js`](lcd-platform-presets.js); **custom** uses min/max swap so portrait = narrow×tall and landscape = wide×short from the current pixel pair. Then `ensureCanonicalSchema` and **`renderScreen`** so the **LCD preview updates immediately**. Pixel readout is **Canvas (pixels)**. **`resolveLcdPresetSelection`** keeps **`lcdPresetId === 'custom'`** as custom (no auto-upgrade to a catalog hit by size). **Widget coordinates are not scaled** on that step — user confirms with **Apply layout** (`applyLayoutRemapToMatchCanvas`).  
   - **Apply layout:** Inspector → Screen → **Apply layout** runs **`applyLayoutRemapToMatchCanvas`** in [`lcd-editor-core.js`](lcd-editor-core.js). It scales layout metrics (`labelColumn`/`valueColumn` widths, paddings, **`menuRowPixelHeight`**, `menuRowGap`) and widget geometry (`x`, `y`, `width`, `height`, `labelWidth`, `valueWidth`, numeric `fontSize`) from **`canvasProfile.remapBaselineWidth` / `remapBaselineHeight`** to the **current** width/height, then sets the baseline to the new size. **`lcdRow` is not remapped** (row slots stay semantic).  
   - **Baseline:** If remap fields are missing, `ensureCanonicalSchema` initializes them to the current canvas size (first Apply is a no-op scale until the canvas changes again).  
   - **Off-screen content:** After a large size/orientation change, some widgets may still sit outside the canvas; the user fixes placement manually for now.  
   - **Row pitch note:** Remap uses `layout.menuRowPixelHeight` or simulator fallback **48** (`SIMULATOR_MENU_ROW_PX_FALLBACK`) to stay aligned with the renderer default until JSON and firmware defaults are unified (see item 11 vs `TSTAT10_MENU_ROW_PX_DEFAULT` **32**).

15. **Background color vs hardware palette (`colorProfile.mode`)**  
   Inspector → Screen background must match what the target can show: **`indexed`** → only **black / white** swatches (1-bit style, e.g. SSD1306); **`reduced_rgb`** → **RGB565** (5-6-5) — theme swatches from `main/LcdTheme.h` via [`lcd-authoring-colors.js`](lcd-authoring-colors.js), optional fine picker that **quantizes** to RGB565. The LCD fill and grid tint use **`resolvedScreenBackgroundCss`** so invalid JSON colors are not rendered as impossible device colors.

16. **Alignment grid (columns / rows) — Inspector → Screen**  
   **`layout.lcdTextColumns`** (default **16**) and **`layout.lcdTextRows`** (default **10**) drive the snap-grid overlay (`injectLcdSnapGridOverlay`) and horizontal **quarter-character** snap step (`canvas width ÷ lcdTextColumns`). Editable in the inspector next to **Snap grid** (clamped **4–48**). They are **per-screen JSON** (saved with the screen), not a separate global pref — keeps authoring portable with the file the device would consume.

Auto-fixes what it can; **errors** block the commit (fix or adjust code). Warnings alone do not fail the hook.  
   - **After clone / `npm install`:** the **`prepare`** script registers Husky hooks (needs this repo to be a **git** checkout).  
   - **Optional in Cursor/VS Code:** install the **ESLint** extension (`dbaeumer.vscode-eslint`). [`.vscode/settings.json`](.vscode/settings.json) runs **Fix all ESLint** when you **save** a file (same `--fix` rules as the hook). Turn that off by removing `source.fixAll.eslint` from `editor.codeActionsOnSave` if you find it noisy. [`.vscode/extensions.json`](.vscode/extensions.json) nudges the ESLint extension for the workspace.  
   - **Manual:** `npm run lint` (check whole tree) or `npm run lint:fix` (fix whole tree).

19. **Firmware color themes (Inspector → Screen)**  
   Canned LCD palettes mirror the firmware **`ThemeList[]`** in **`main/lcdTheme.c`** (ESP32 repo `T3-programmable-controller-on-ESP32`): same **10** entries and **RGB565** field order (`text`, `backgroundColor`, `backgroundColor1`, …). Simulator module **[`lcd-firmware-color-themes.js`](lcd-firmware-color-themes.js)** maps:  
   - **`backgroundColor` (index 1)** → **`styles.bg`** (LCD fill, quantized in **`reduced_rgb`** per [`lcd-authoring-colors.js`](lcd-authoring-colors.js)).  
   - **`backgroundColor1` (index 2)** → **`styles.highlight`** (menu-row selection band in `network-settings-renderer.js`).  
   - **`colorProfile.themeTokens`** — sets **`bg`** and **`accent`** (highlight) so `ensureCanonicalSchema` in [`lcd-editor-core.js`](lcd-editor-core.js) stays aligned.  
   Choosing a theme sets **`colorProfile.mode`** to **`reduced_rgb`** so colors match hardware 5-6-5 steps (user can switch back to **`indexed`** manually if needed).  
   **UI:** `#editor-lcd-firmware-theme` in **`Tstat10.html`**; options populated in **`bindEditorControls`** in [`network-settings-renderer.js`](network-settings-renderer.js). **`Custom`** = current **`styles.bg` / `styles.highlight`** do not match any canned pair (`findMatchingFirmwareThemeId`).  
   **Project-wide:** Applying a theme calls **`propagateProjectWideFirmwareTheme`**, which updates every path in **`PROJECT_SCREEN_JSON_PATHS`** that already has a **`localStorage`** draft (same idea as project-wide **Background**); uncached paths are fetched and cached in the background. Changing **Background** alone still uses **`propagateProjectWideBackground`**; if the screen already has **`styles.highlight`**, **`commitProjectWideBackgroundIfPossible`** can merge **bg + highlight + mode** via the firmware-theme merge path.  
   **Ground truth:** full theme struct and names live in **`lcdTheme.c`** / **`LcdTheme.h`**; the web sim only applies the two fields above plus mode/tokens until per-widget color mapping is needed.

20. **Fixed LCD size and position across simulator pages**  
   The product has **one** logical framebuffer (**240×320** portrait, ILI9341). In the hosted simulator, **`enforceSimulatorFixedLcdCanvas`** in [`network-settings-renderer.js`](network-settings-renderer.js) runs on every **`renderScreen`** (after **`ensureCanonicalSchema`**) and sets **`canvasProfile.width` / `height`**, **`layout.lcdCanvas`**, **`layout.canvas`**, **`layout.orientation`**, and **`canvasProfile.lcdPresetId`** (**`tstat10_fw_240`**) so **no route** can shrink, grow, or rotate the LCD slot relative to another. Stale **`localStorage`** drafts or odd JSON cannot change the preview geometry when navigating between Home, Setup, Wi‑Fi, etc.  
   **Workbench nudge** (LCD translate inside the bezel) is **global** only: **`ui-bridge.js`** persists **`tstat10_lcd_bezel_nudge_px`** and applies **`window._applyTstatLcdNudgeTransform`**. The simulator **does not** re-read **`canvasProfile.previewOffsetX/Y`** from each screen’s cached JSON when switching **`jsonPath`** (removed **`applyWorkbenchNudgeFromLoadedScreenData`**); **`syncWorkbenchNudgeIntoScreenDataForCache`** still writes the **current** globals into the active draft so caches stay consistent. Inspector → **LCD outline** hint states that the offset is shared across pages.
